# 安全モード実装計画

## 1. 目的

ユーザーが意図せず`~/GeminiCLI`ディレクトリ外のファイルを操作したり、`rm`のような破壊的なコマンドを実行したりすることを防ぐための安全機能を実装する。システムの堅牢性を高め、より安全な操作環境を提供する。

## 2. 実装方針

バックエンド (`server.js`) にコマンドインターセプターを実装し、クライアントから送信されたコマンドを実行前に検証・加工する。フロントエンドで設定のON/OFFを可能にするが、権限チェックはすべてバックエンドで強制する。

## 3. 実装タスク（バックエンド: `server.js`）

### 3.1. シェルセッションの初期化
-   [ ] `node-pty`でシェルをスポーンする際、初期カレントワーキングディレクトリを`~/GeminiCLI`に設定する。

### 3.2. 堅牢なパス検証機能の実装
-   [ ] `path`モジュールを活用し、パス トラバーサル攻撃に耐性のある`isPathSafe`関数を実装する。
    -   `path.resolve`と`path.relative`を組み合わせて、パスが`~/GeminiCLI`内に留まることを保証する。

### 3.3. コマンドインターセプターの実装
-   [ ] WebSocketでメッセージを受信後、`pty`に書き込む前にすべてのコマンドを検証する処理層を追加する。
-   [ ] **`cd`コマンドの制御:**
    -   `cd`コマンドを特別扱いし、移動先のパスを`isPathSafe`で検証する。
    -   許可されていないディレクトリへの移動はブロックし、ユーザーに通知する。
-   [ ] **汎用的なパス検証:**
    -   正規表現を使用し、コマンド文字列全体からファイルパスの可能性がある引数（リダイレクト先などを含む）をすべて抽出する。
    -   抽出したすべてのパスを`isPathSafe`で検証し、一つでも違反があればコマンド全体をブロックする。
-   [ ] **`rm`コマンドの無効化:**
    -   コマンドが`rm`で始まる場合、実行をブロックする。
    -   代わりに`trash`コマンドの使用を促すメッセージをクライアントに送信する。
-   [ ] **`study_log.db`の保護:**
    -   コマンドに`study_log.db`が含まれ、かつ`manage_log.py`が含まれていない場合、実行をブロックする。
    -   `manage_log.py`経由での操作を促すメッセージを送信する。

### 3.4. 安全モードの状態管理
-   [ ] 各WebSocket接続 (`ws`) ごとに`isSafeMode`フラグを持たせ、デフォルトで`true`に設定する。
-   [ ] フロントエンドから`setSafeMode`メッセージを受信したら、対応する接続のフラグを更新する。

## 4. 実装タスク（フロントエンド）

### 4.1. 設定UIの実装 (`rescue-console-app/components/rescue-console/RescueConsoleApp.tsx`)
-   [ ] `SettingsPanel`コンポーネントに「安全モード」のON/OFFを切り替えるトグルスイッチを追加する。
-   [ ] `RescueConsoleApp`コンポーネントに`safeMode`の状態 (`useState`) を追加する。
-   [ ] トグルスイッチの変更をハンドリングし、`safeMode`の状態を更新する関数 (`handleSafeModeChange`) を実装し、`SettingsPanel`に渡す。

### 4.2. バックエンドへの通知
-   [ ] `handleSafeModeChange`関数内で、WebSocketを通じて`{ type: 'setSafeMode', enabled: boolean }`形式のメッセージをバックエンドに送信する処理を実装する。
    -   **課題:** WebSocketインスタンスは各ウィンドウが個別に保持しているため、設定変更をすべてのウィンドウ（またはグローバル）に通知する仕組みを検討する必要がある。まずはアクティブなウィンドウから通知する形で実装を進める。

## 5. 開発ワークフロー

1.  この計画書 (`current_task_handover.md`) を作成する。
2.  バックエンドの`isPathSafe`関数から実装を開始する。
3.  機能単位で細かくコミットを行う (`feat:`, `fix:`など)。
4.  バックエンドのコマンドインターセプト機能を一通り実装する。
5.  フロントエンドのUIを実装する。
6.  WebSocketでの通信を実装し、フロントとバックを連携させる。
7.  各機能が正しく動作するか、テストケースを想定して検証する。

この計画に従い、慎重かつ段階的に開発を進める。
